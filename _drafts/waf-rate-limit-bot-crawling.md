운영 중인 한 뉴스 사이트에서 RDS CPU가 99%까지 치솟는 장애가 세 번째 반복됐다. 매번 같은 패턴이었는데 이번에 WAF Rate-based Rule로 대응한 과정과, 그 과정에서 만난 함정들을 정리한다.

## 증상

- 특정 시간대에 RDS CPU 13% → 96.7% (max 99.98%), DB 커넥션 최대 1,027개
- 애플리케이션 응답 지연 → 사실상 서비스 장애
- 몇 달 간격으로 세 번째 반복

## 원인 분석 1 — WAF 로그: 범인은 봇넷

WAF 전수 로그(CloudWatch Logs, KEEP 액션 포함)를 분석해 보니 장애 시간대 50분 동안 기사 조회 경로로 **14만 건 이상**이 origin까지 통과하고 있었다.

특징이 명확했다:

- **IP는 사건마다 전부 교체된다.** 수천 개의 해외 IP가 로테이션되기 때문에 특정 IP 블랙리스트는 의미가 없다
- 대신 **반복되는 건 위조된 User-Agent 지문**이었다. 존재할 수 없는 브라우저 버전 조합(예: 구식 macOS `10_15_7` 고정 + 최신 Chrome)이 전체 통과 트래픽의 56% 이상
- 기사 번호를 순차 증가시키며 아카이브를 전수 수집하는 전형적인 크롤링 봇넷

## 원인 분석 2 — Performance Insights: 증폭기는 쿼리

트래픽은 트리거일 뿐이고, DB를 실제로 죽인 건 캐싱 없는 고비용 쿼리였다. Performance Insights로 피크 시간대를 실측하니:

- 인기 키워드 위젯 쿼리 (`REGEXP '[가-힣]'` + `ORDER BY RAND()`) — 부하의 43%
- 인물 검색 초성 쿼리군 — 부하의 46%

두 쿼리가 부하의 90%를 차지했다. `ORDER BY RAND()`는 매 호출마다 풀스캔+정렬이라 봇이 페이지를 긁을 때마다 그대로 증폭된다. **근본 해결책은 쿼리 캐싱(애플리케이션 개선)이고, WAF는 완화책**이라는 걸 먼저 인정하고 시작해야 한다.

## 대응 설계 — Rate-based Rule

기사 조회 경로에 scope-down을 걸고 Rate-based Rule을 설계했다.

**임계치 선정: 100 vs 200 (2분 기준)**

2주치 로그로 각 임계치에 걸리는 IP를 전수 분류해서 비교했다:

- 200으로 하면 악성 봇넷 4개 중 3개가 빠져나간다. 그런데 검색엔진 봇은 여전히 걸린다
- 100으로 하면 봇넷은 다 걸리지만, **Googlebot(2분에 148건)과 Bingbot도 같이 걸린다**

즉 어느 쪽이든 검색엔진이 차단되므로, 답은 "낮은 임계치 + 검색엔진 화이트리스트 선등록"이었다. 뉴스 사이트에서 Googlebot 차단은 SEO 타격 = 매출 타격이라 절대 조건이다.

## 함정 1 — 검색엔진 화이트리스트, 어디까지 넣어야 하나

Google/Bing은 공식 크롤러 IP 목록을 JSON으로 제공한다:

- Googlebot: `googlebot.json`
- Bingbot: `bingbot.json`

여기까지는 다들 아는데, 놓치기 쉬운 게 있다. **`special-crawlers.json`** — Mediapartners-Google(애드센스 크롤러) 등이 여기에 들어있다. 광고를 게재하는 사이트라면 이걸 빠뜨리는 순간 애드센스 크롤러가 rate limit에 걸리고 광고 수익에 영향이 갈 수 있다. 실제로 COUNT 모드 모니터링 중에 이 누락을 발견해서 보강했다.

참고: [Verifying Googlebot 공식 문서](https://developers.google.com/search/docs/crawling-indexing/verifying-googlebot)

IP CIDR 화이트리스트는 목록이 수시로 바뀌어 관리 부담이 있다. 장기적으로는 AWS WAF **Bot Control**의 `VERIFIED_BOT` 레이블로 공식 봇을 자동 판별하는 게 더 견고하다 (다만 유료 관리형 규칙이라 비용 검토 필요).

## 함정 2 — 바로 BLOCK 하지 말 것

Rate-based Rule은 반드시 **COUNT 모드로 먼저 배포**하고 실제 로그로 오탐을 검증한 뒤 BLOCK으로 전환해야 한다. 이번에 COUNT 기간 동안 발견한 것들:

- **전파 지연**: 화이트리스트(IP set) 변경 직후 약 40분간 일부 요청이 화이트리스트를 타지 못했다. CloudFront에 연결된 WAF는 전역 엣지로 전파되는 시간이 있다. 변경 직후의 로그만 보고 "화이트리스트가 안 먹네?"라고 판단하면 안 된다
- **꼬리 트래픽 검증**: 카운트에 1~2건 걸린 애매한 IP들은 타임스탬프를 재조회해 보니 대부분 "몇 시간 간격으로 몰아치는" 크론 스케줄 패턴이었다. 정상 사용자가 아니라는 근거를 수치로 확보
- **정상 사용자 실측**: 브라우저 자동화(Playwright)로 실제 페이지에 접속해서 네트워크 요청을 전수 캡처했다. 정상 사용자는 페이지 1회 열람에 해당 경로 요청이 정확히 1건 — 2분에 100건 임계치로는 오탐이 나올 수 없음을 클라이언트 관점에서 교차 검증했다

## 결과

BLOCK 전환 후 전날 동일 시간대와 비교:

| 지표 | 전환 전 | 전환 후 |
|---|---|---|
| 서버 도달 비율 | 81.5% | 58.1% |
| WAF 차단 비율 | 18.5% | 41.9% |

서버 도달 트래픽 약 41% 감소, 검색엔진 봇 정상 통과, 정상 사용자 오탐 0건. 롤백도 규칙을 COUNT로 되돌리기만 하면 되므로 다운타임 없이 즉시 가능하다.

## 배운 것

1. **봇넷 시대에 IP 블랙리스트는 죽었다.** IP는 로테이션되고, 남는 건 UA 지문 같은 행동 패턴이다
2. **WAF는 완화책, 근본은 애플리케이션.** 고비용 쿼리 캐싱 없이는 다음 봇넷에 또 당한다
3. **Rate Limit 설계는 "누가 걸리는가"의 전수 분석이 먼저다.** 임계치를 감으로 정하면 Googlebot을 차단하는 대형 사고가 난다
4. **COUNT → 검증 → BLOCK 절차는 생략하면 안 된다.** 전파 지연, 애매한 꼬리 트래픽, 화이트리스트 누락은 전부 COUNT 기간에 발견됐다
